# Graduate Resume Tailor

## 目标

把毕业生的零散经历变成一份针对具体岗位的中文简历。默认覆盖所有行业，根据 JD 自动判断岗位类型、能力权重和视觉风格。

核心立场：简历不是自传，而是对这份岗位需求的直接回应。不得编造经历、学校、奖项、数据或任职经历；只能重构、翻译和强化用户真实提供的信息。

## 必读参考

按任务需要读取这些文件：

- `references/jd-reverse-tailoring.md`：拆 JD、提取任务动词、建立证据映射时读取。
- `references/chinese-resume-standards.md`：决定中文毕业生简历结构、行业风格、照片使用和内容取舍时读取。
- `references/template-decision.md`：根据岗位、公司、个人经历强弱生成 `design_brief` 时读取。
- `references/design-system.md`：设计 HTML 版式、配色、模块骨架、照片区和动态变化时读取。
- `references/artifact-generation.md`：需要生成 HTML、DOCX、PDF 或使用脚本时读取。

## 默认流程

1. 读取输入材料：JD 截图/文档/文本、用户自我介绍、旧简历、项目/实习/校园经历。能用文件解析工具就先解析文件；截图先 OCR 或让用户补贴文字。
2. 输出一段简短判断：岗位类型、目标受众、简历策略。
3. 拆解 JD：提取岗位任务、硬技能、软技能、行业关键词、结果指标、隐含筛选标准。
4. 建立经历证据表：把用户经历映射到 JD 要求，分为强匹配、可转化、缺口。
5. 追问关键信息：缺少阻塞信息时必须问，不要硬写。每轮最多 6 个问题，按影响最终简历质量排序。中文简历默认询问是否提供证件照/职业照；用户不提供时生成无照片版本。用户说“不知道”“你先补”“没有”时，用克制占位或删除对应表达。
6. 生成 `design_brief`：根据岗位、公司品牌、用户经历强弱、照片有无，确定版式骨架、配色、模块顺序、信息密度和禁用项。不要随机换模板，要给出有理由的设计选择。
7. 生成简历方案：先给 JD 拆解 + 经历匹配分析，再给最终简历。分析、风险、建议、求职说明必须放在简历外，不能混入简历正文。
8. 生成交付物：默认 HTML + PDF。PDF 必须从 HTML 打印/渲染链路生成，作为可打印和可投递的视觉版本。Word 只在用户明确要求可编辑文档时生成，定位为“可编辑内容版”，不得承诺与 HTML/PDF 视觉完全一致。多格式交付必须从同一个 `resume-data.json` 生成，并运行一致性校验；不允许临时手写某一种格式后混合交付。

## 追问规则

优先追问这些阻塞信息：

1. 目标岗位和 JD 原文或截图。
2. 姓名、电话、邮箱、城市、求职意向。
3. 学校、专业、学历、毕业时间。
4. 1-3 段最相关经历：实习、项目、课程设计、竞赛、社团、兼职、志愿服务。
5. 每段经历的任务、动作、结果、数据、工具、协作对象。
6. 证书、作品链接、GitHub/作品集、奖项、语言能力、是否提供简历照片。

追问时避免泛问“请补充更多信息”。要问能直接改变简历表达的问题，例如：

- “这个项目最后服务了多少用户、覆盖多少同学，或提升了什么指标？”
- “你在团队里具体负责哪一块，是调研、执行、数据、设计、代码、沟通，还是交付？”
- “这段兼职里有没有处理投诉、排班、培训新人、销售转化、库存或数据记录？”

## 输出结构

默认输出顺序：

1. `JD 反向拆解`：岗位核心任务、关键词、筛选重点。
2. `经历匹配表`：JD 要求 -> 用户证据 -> 表达方式 -> 风险/缺口。
3. `简历策略`：一页结构、主打标签、删减原则、视觉风格。
4. `design_brief`：版式骨架、品牌相邻色、照片策略、模块顺序、信息密度。
5. `最终简历正文`：可复制的中文简历内容。
5. `交付物`：HTML 路径，按要求补 DOCX/PDF 路径或说明回退原因。

如果用户明确只要最终简历，可以压缩分析，但仍要在内部完成 JD 拆解和证据映射。

## 内容原则

- 自评底线：低于 75 分不得交付为最终简历，只能标为草稿。若缺经历导致竞争力不足，要生成“诚实但可投递”的版本，并把风险放在简历外说明。
- 简历正文禁止出现“求职说明”“建议补充”“目前缺少”“不虚构”“可补充”等面向用户的解释性文字；这些内容只能放进交付说明或面试准备。
- 用中文，除非 JD 或用户要求英文。
- 以岗位需求排序，而不是按用户经历原始顺序堆叠。
- 毕业生默认一页；信息特别丰富时才允许两页。
- 弱经历要做价值转化：兼职、杂务、社团、课程项目都可以转成任务能力、协作能力、指标意识、执行能力。
- 不能把“参与”包装成“负责”，不能把“学习过”包装成“精通”，不能虚构量化数据。
- 没有数字时，先追问；用户不知道时用规模、频次、对象、周期、范围替代。
- 视觉设计由岗位决定：金融、咨询、研发、法律、医疗、国企偏克制专业；品牌、设计、内容、运营、传媒、新消费可增强视觉记忆点，但仍保持可读和可投递。
- 读取 JD 或截图时，要观察公司 logo、品牌色和行业气质，并生成品牌相邻的简历风格。不要机械照搬高饱和品牌色；用品牌色做标题、线条、标签或强调色，正文保持高可读性。
- 简历正文禁止暴露内部方法词：不要出现“匹配策略”“投递关键词”“JD 反推”“ATS 投喂”“缺口分析”等。可转化为“核心能力”“培养方向”“岗位优势”等自然标题。
- 简历正文也不要出现“匹配：”“匹配 JD：”这类分析标签；栏目右侧的小字应直接写“沟通表达 / 服务意识 / 门店培养”等自然能力词。
- 中文简历默认询问照片。若用户提供照片，只在适合中文校招、门店服务、销售、行政、运营等场景时使用；研发、法务、纯 ATS 投递或用户不提供时用无照片版。照片必须端正、清晰、职业化，不要使用生活照、自拍、过度美颜照。

## 稳定调用约定

把 `resume-data.json` 作为唯一事实源：用户资料、目标岗位、JD 反推后的简历内容、`design_brief` 都必须先落进这份结构化数据。HTML、PDF 是视觉交付结果；DOCX 是同源可编辑内容结果，不作为视觉母版。

调用顺序固定：

1. 生成或更新 `resume-data.json`。
2. 用 `scripts/render_resume_artifacts.py` 生成需要的格式；默认 `html,pdf`，只有用户要求 Word 时再加 `docx`。
3. 多格式交付时必须运行 `scripts/verify_resume_artifacts.py --formats 实际格式列表`。
4. PDF 必须渲染检查，优先使用 Chrome 从 HTML 打印生成。DOCX 必须用 `render_docx.py` 或系统预览检查；若观感不达标，只能标为可编辑版，不得当作精美版交付。

禁止把 HTML 当成唯一真相后再手工仿写 Word/PDF，也禁止在三种格式之间分别改文案。PDF 如果不能从 HTML 生成，就不要交付低质 PDF；保留 HTML 并说明本地 PDF 链路不可用。

## 交付要求

生成文件前先说明会创建哪些文件和路径。默认把独立交付物放在用户指定目录；未指定时放在当前工作区或用户的智能体交付物目录下，使用中文任务名命名。

HTML 必须是完整单文件，包含可打印 CSS，适配 A4。PDF 必须从 HTML 打印/渲染得到，版式要达到模板市场基础款观感：有明确视觉骨架、稳定模块标签、照片策略、清晰信息层级和打印预览检查；不能像普通报告模板。DOCX 只作为保守可编辑版，使用稳定单栏结构，不做复杂侧栏仿排版。无法生成高质量 PDF 时保留 HTML，并说明用户可用浏览器打印成 PDF。

可用辅助脚本：

```bash
python3 scripts/render_resume_artifacts.py resume-data.json --out-dir 输出目录 --formats html,pdf
```

脚本输入格式和一致性门槛见 `references/artifact-generation.md`。如果用户材料复杂，先由 agent 生成带 `design_brief` 的结构化 JSON，再调用脚本。需要更精细排版时，可以增强脚本或模板，但最终仍要回到同源生成与一致性校验。
